# Orchestra: ТЗ и план MVP

## 1. Цель и контекст

**Orchestra** – сервис‑оркестратор для работы LLM с кодовой базой проекта. На уровне MVP задача проста:

> Иметь консольный инструмент, который умеет читать проект, формировать контекст для LLM и применять предлагаемые моделью изменения в файлах (с возможностью dry‑run).

Дальнейшая эволюция: вынесение логики в daemon, индексация кода, гибридный поиск, agent loop, интеграция с IDE.

---

## 2. Общие требования к MVP

### 2.1. Технологический стек

* Язык: **Go** (1.22+).
* Сборка: один бинарник CLI.
* Взаимодействие с LLM:

  * HTTP API совместимый с OpenAI (локальный vLLM или внешний провайдер).
  * Конфигурируемые: `API_BASE`, `API_KEY`, `MODEL`.
* ОС: Linux / Windows (минимум поддержка двух платформ).

### 2.2. Целевая архитектура (high‑level)

На уровне целевой архитектуры (после MVP):

* **orchestra-daemon**

  * долгоживущий процесс;
  * хранит индекс файлов и кеш;
  * предоставляет gRPC/HTTP API для CLI/IDE.
* **orchestra-cli**

  * тонкий клиент;
  * отправляет задачи, показывает план и патчи, применяет изменения по запросу пользователя.

На уровне **MVP v0.x** допускается:

* один бинарник `orchestra`, работающий в режиме:

  * *простой CLI* без постоянного демона (Спринт 1–2);
  * последующим добавлением режима `orchestra daemon` (Спринт 3).

## 3. Функциональные требования MVP (v0.1–v0.3)

### 3.1. MVP v0.1 — базовый консольный агент

**Цель:** от запроса пользователя до изменения файлов через LLM, без индекса и сложного поиска.

Функции:

1. **Инициализация проекта**

   * Команда: `orchestra init`.
   * Создает файл конфигурации `.orchestra.yml` в корне проекта.
   * Содержимое конфига (минимум):

     * `project_root`;
     * список исключаемых директорий (`.git`, `node_modules`, `dist`, `build`, и т.д.);
     * настройки LLM (`api_base`, `model`, `temperature`, `max_tokens`).

2. **Режим запроса кода (analyze/apply)**

   * Команда: `orchestra apply "описание задачи"`.
   * Шаги:

     1. Определение корня проекта (текущая директория или `project_root`).
     2. Сканирование файлов по маскам/исключениям.
     3. Формирование промпта:

        * краткое описание структуры проекта (список файлов с путями);
        * содержимое *ограниченного набора файлов* (см. ниже);
        * запрос пользователя.
     4. Вызов LLM.
     5. Парсинг ответа модели.
     6. Применение изменений.

3. **Ограниченный контекст файлов**

   * На этапе MVP v0.1 без RAG, но с простыми эвристиками:

     * если пользователь явно указал файл: включить этот файл целиком;
     * если команда выполняется из поддиректории: приоритизировать текущий путь;
     * лимит на суммарный объем кода в промпте (например, 30–50 KB).

4. **Формат ответа модели**

   Для MVP вводим простой текстовый формат, удобный для парсинга:

   ```
   ---FILE: path/to/file.go
   <<<BLOCK
   // старый код (для поиска, опционально)
   >>>BLOCK
   // новый код (полный блок/функция)
   ---END
   ```

   Допустимые типы операций на уровне v0.1:

   * `replace_block`: заменить найденный блок кода новым блоком (по текстовому поиску или простому fuzzy‑поиску);
   * `replace_file`: перезаписать файл целиком (опционально, отдельный режим).

5. **Применение изменений**

   * По умолчанию режим **dry‑run**:

     * показывать список затрагиваемых файлов;
     * показывать diff по каждому файлу.
   * Флаг `--apply`:

     * физически вносит изменения в файлы;
     * сохраняет бэкап оригинала с суффиксом `.orchestra.bak` (один уровень бэкапа).

6. **Логирование и ошибки**

   * Лог в stdout в человекочитаемом виде.
   * Флаг `--debug` для подробного вывода (HTTP‑запросы к LLM без тела, состояние парсинга, информация о файлах).

**Ограничения v0.1:**

* Нет постоянного индекса и поиска.
* Нет интеграции с git (только бэкапы файлов).
* Нет агентного цикла и инструментов типа `run tests`.

---

### 3.2. MVP v0.2 — поиск и чуть более умный контекст

**Цель:** добавить минимальный, но полезный поиск и планирование.

Дополнительные функции:

1. **Минимальный поиск по коду**

   * Лексический поиск в файлах:

     * по имени файла;
     * по простому текстовому совпадению (строка/подстрока);
   * Команда: `orchestra search "login handler"`.

2. **Автоподбор контекста для **``

   * Перед вызовом LLM:

     * выполнить текстовый поиск по ключевым словам из запроса пользователя;
     * добрать N наиболее релевантных файлов/фрагментов в промпт;
     * обязательно включать:

       * текущий файл (если запущено внутри файла/директории);
       * явно указанные пользователем файлы.

3. **План изменений**

   * При `orchestra apply` модель сначала возвращает **план**:

     * список шагов: какие файлы трогать, какие изменения вносить;
     * этот план показывается пользователю перед генерацией кода.
   * Режим: `--plan-only` для генерации плана без изменения кода.

4. **Базовая интеграция с git**

   * Проверка «грязности» репозитория:

     * если есть незакоммиченные изменения, предупреждать пользователя;
   * Опциональный флаг `--git-commit`:

     * после успешного применения изменений создать коммит с сообщением, сгенерированным из задачи (`feat(orchestra): ...`).

---

### 3.3. MVP v0.3 — daemon‑режим и кеш

**Цель:** минимальный переход к целевой архитектуре без сложного RAG.

Дополнительные функции:

1. **Daemon‑режим**

   * Команда: `orchestra daemon`.
   * Функции:

     * хранит список файлов и их хэш/mtime в памяти;
     * обслуживает простые запросы по локальному gRPC/HTTP API:

       * `ListFiles`;
       * `ReadFile`;
       * `SearchText`.

2. **CLI поверх daemon**

   * При запуске `orchestra apply`:

     * сначала пытается подключиться к локальному daemon;
     * если тот недоступен – работает в режимах v0.1–v0.2 (fallback).

3. **Кеш структуры проекта**

   * Один файл кеша в `.orchestra/cache.db` или подобный;
   * хранение:

     * список файлов;
     * их размеры, mtime;
     * базовая информация для быстрого сканирования.

4. **Подготовка к RAG**

   * На этом этапе достаточно API вида `GetContext(suggestion)`:

     * возвращает список файлов/фрагментов по простым эвристикам;
     * сама логика RAG пока минимальна, без эмбеддингов.

## 4. Нефункциональные требования

1. **Производительность**

   * Сканирование проекта (до ~5–10k файлов) в v0.1–v0.2 должно укладываться в несколько секунд.
   * В daemon‑режиме повторные операции `apply` не должны заново обходить файловую систему целиком.

2. **Надёжность**

   * Любые изменения в файлах должны быть либо откатываемы (бэкапы, git), либо документированы.
   * При ошибке парсинга ответа LLM – не вносить изменений и отдавать понятную ошибку.

3. **Расширяемость**

   * Архитектура кода должна позволять добавлять:

     * новые «инструменты» агента (run tests, search symbols, run linter);
     * новые стратегии формирования контекста;
     * новые режимы генерации (diff, AST‑операции).

## 5. Риски и ограничения MVP

1. **Формат ответа LLM**

   * Один формат: `---FILE / <<<BLOCK / >>>BLOCK / ---END`.
   * Политика: при нарушении формата **изменения не применяются**, инструмент падает с ошибкой.

2. **Поиск блока для замены**

   * Для v0.1: **только точный текстовый поиск** по старому блоку.
   * Fuzzy-поиск: явно отнесён в будущие версии.

3. **Лимит контекста**

   * Есть лимит (например, 50 KB).
   * Лимит **обязательно конфигурируется** в `.orchestra.yml`.

4. **Конфликты изменений**

   * Если несколько изменений затрагивают пересекающийся диапазон строк/блоков в одном файле → ошибка «конфликт изменений, ручное разруливание».

## 7. План по спринтам

### Спринт 1 (v0.1) — «Работающий CLI‑агент»

**Цель:** от текстового запроса до изменения файлов, без индекса и поиска.

**Задачи:**

1. Инфраструктура проекта

   * Создать репозиторий `orchestra`.
   * Настроить базовый layout на Go (cmd/pkg/internal и т.п.).
   * Добавить конфигурацию (viper или свой простой парсер YAML).

2. Базовый CLI

   * Команда `orchestra init`.
   * Команда `orchestra apply` (пока без плана и поиска).

3. Взаимодействие с LLM

   * Модуль клиента OpenAI‑совместимого API.
   * Структура промпта для v0.1.
   * Структура ожидаемого ответа (парсер формата `---FILE/...`).

4. Применение изменений

   * Модуль применения операций `replace_block` и `replace_file`.
   * Реализация dry‑run и `--apply`.
   * Сохранение `.orchestra.bak` для затронутых файлов.

5. Логирование и ошибки

   * Единый логгер.
   * Чёткие сообщения об ошибках при парсинге ответа.

**Критерий готовности:**

* В небольшом тестовом проекте можно выполнить `orchestra apply "добавь логирование в main.go"` и получить измененный файл с логикой.

---

### Спринт 2 (v0.2) — «Поиск и план»

**Цель:** сделать выбор контекста умнее и безопаснее.

**Задачи:**

1. Поиск

   * Реализовать текстовый поиск по файлам (`orchestra search`).
   * Интегрировать поиск в `apply` для подбора релевантных файлов.

2. План изменений

   * Изменить протокол общения с LLM:

     * сначала запрос на план (список файлов и действий);
     * затем запрос на кодовые изменения.
   * CLI‑режим `--plan-only`.

3. Git‑интеграция (минимум)

   * Проверка статуса репозитория (git status).
   * Предупреждение при незакоммиченных изменениях.
   * Опция `--git-commit` для автоматического коммита после `--apply`.

**Критерий готовности:**

* Инструмент показывает план того, что будет изменено.
* Пользователь может видеть, что и где будет тронуто, до генерации кода.

---

### Спринт 3 (v0.3) — «Daemon и кеш»

**Цель:** перейти к целевой модели с долгоживущим процессом.

**Задачи:**

1. Daemon

   * Реализовать режим `orchestra daemon`.
   * Локальный gRPC или HTTP API.

2. CLI поверх daemon

   * `orchestra apply` сначала пытается работать через daemon.
   * При отсутствии daemon – fallback к режиму v0.2.

3. Кеш структуры проекта

   * Простая БД/файл кеша для хранения списка файлов и метаданных.
   * Инвалидация кеша по mtime и изменению размера файлов.

4. Подготовка к RAG

   * Вынести логику выбора контекста в отдельный модуль.
   * Сделать API, пригодное для будущей интеграции с эмбеддингами и векторным поиском.

**Критерий готовности:**

* При повторных запусках на том же проекте `orchestra apply` работает заметно быстрее за счет кеша/daemon.

## 9. Открытые вопросы

* **Поддерживаемые языки на старте:**

  * **Любые текстовые файлы**, без гарантий синтаксической валидации.
  * Отдельно: «Go имеет спец-хендлинг (go fmt / go vet) в будущем».

* **Поведение при падении LLM/API:**

  * Если LLM недоступен → понятная ошибка и никакие файлы не трогаем.
  * Если LLM вернул формат, который не распарсился → показываем сырое тело, ничего не применяем.

* **Валидация синтаксиса:**

  * Для v0.1: **нет жёсткого требования**.
  * Для Go можно добавить в v0.2/0.3 `go fmt` + `go test` как отдельный флаг.
